# AY-3-8910 / YM2149F State Machine

> This page documents how the AY chip emulator works in the VTIR codebase —
> what state it keeps, how it advances each clock tick, and how it connects
> to the audio output pipeline.  The reference implementation is
> `legacy/AY.pas`; the Rust port lives in `crates/vti-ay/`.

---

## What is the AY-3-8910?

The **AY-3-8910** (and its compatible sibling the **YM2149F**) is a
*Programmable Sound Generator* (PSG) chip manufactured from the late 1970s.
It produced the characteristic "chiptune" sound of the ZX Spectrum, Amstrad
CPC, MSX, and many other 1980s home computers.

The chip provides:
- **Three independent square-wave tone channels** (A, B, C)
- **One shared noise generator** with a 17-bit LFSR
- **One hardware envelope generator** with eight distinct shapes
- **A mixer** that independently routes tone and noise into each channel
- **Per-channel amplitude control** — either a fixed 4-bit level or
  envelope-modulated amplitude

The chip has **16 write-only 8-bit registers** (R0–R15).  VTIR mirrors these
in the `AyRegisters` struct and computes a fresh snapshot every tracker frame
(50 Hz).

## Hardware registers

| Register | Bits | Field |
|----------|------|-------|
| R0, R1   | 12   | Channel A tone period (R0 = low 8, R1 = high 4) |
| R2, R3   | 12   | Channel B tone period |
| R4, R5   | 12   | Channel C tone period |
| R6       | 5    | Noise period |
| R7       | 8    | Mixer control (see below) |
| R8       | 5    | Channel A amplitude / envelope enable |
| R9       | 5    | Channel B amplitude / envelope enable |
| R10      | 5    | Channel C amplitude / envelope enable |
| R11, R12 | 16   | Envelope period (R11 = low, R12 = high) |
| R13      | 4    | Envelope shape selector (0–15) |

### Mixer register (R7) bit field

| Bit | Meaning (0 = enabled) |
|-----|----------------------|
| 0   | Tone A enable |
| 1   | Tone B enable |
| 2   | Tone C enable |
| 3   | Noise A enable |
| 4   | Noise B enable |
| 5   | Noise C enable |

### Amplitude registers (R8–R10) bit field

| Bits | Meaning |
|------|---------|
| 3:0  | Fixed amplitude level (0–15) |
| 4    | `1` = ignore fixed level; use envelope amplitude instead |

## `SoundChip` state (chip.rs)

The `SoundChip` struct maintains a complete snapshot of one chip's runtime
state.  Key fields:

```
Tone counters     ton_counter_a/b/c   u32   16.16 fixed-point; wrap triggers toggle
Tone outputs      ton_a/b/c           i32   Current square-wave bit (0 or 1)
Noise counter     noise_counter       u32   Divider counter
Noise seed        noise_seed          u32   17-bit LFSR state
Noise output      noise_val           u32   LSB of noise_seed
Envelope counter  envelope_hi/lo      u32   64-bit split; hi overflows → step
Envelope output   ampl                i32   Current envelope amplitude (0–31)
Envelope shape    env_shape           EnvShape
Envelope phase    first_period        bool  First half of period flag
```

The `registers` field shadows the 14 hardware registers that the tracker
writes each frame.

## Noise LFSR

```
Function: noise_generator(seed: u32) -> u32
Taps: bit 13 ⊕ bit 16
```

```rust
let bit = ((seed >> 13) ^ (seed >> 16)) & 1;
let s   = ((seed << 1) & 0x1_FFFF) | 1;
s ^ bit
```

- Operates on a 17-bit value (bits 0–16 of the u32).
- Called once per noise clock cycle (every `noise_period` AY clocks).
- `noise_val = noise_seed & 1` is the output bit mixed into channels.
- Period ≈ 131,071 unique states before the sequence repeats.

The Rust output is verified against `crates/vti-ay/tests/fixtures/pascal-baselines/noise_lfsr.json`
— 200 steps generated by the original Pascal code.

## Tone generators

Each channel runs a simple **divide-by-N counter** that toggles the output
bit when it reaches the programmed period:

```
hi_a = ton_counter_a >> 16        // extract integer part of counter
hi_a += 1
if hi_a >= ton_a_period:
    hi_a -= ton_a_period
    ton_a ^= 1                    // toggle: 0→1 or 1→0
ton_counter_a = hi_a << 16
```

This produces a **50 % duty-cycle square wave** at:

```
f_tone = AY_CLOCK / (2 × period)
```

With the default 1,773,400 Hz clock and period = 1 the maximum tone
frequency is about 886 kHz; practical notes use periods from ~14 (B-8) up to
~3,840 (C-1).

## Envelope generator

### Shapes

Envelope type R13 maps to one of eight distinct shapes (not 16 — values 0–3
all behave like "decay hold 0", 4–7 like "attack hold 31"):

| Type | Name | Description |
|------|------|-------------|
| 0–3  | Hold0 | Decay 31→0, hold at 0 forever |
| 4–7  | Hold31 | Attack 0→31, hold at 31 forever |
| 8    | Saw8 | Continuous sawtooth down: 31→0→31→0… |
| 9    | Hold0 | Same as types 0–3 |
| 10   | Triangle10 | Triangle down-up: 31→0→31→0… |
| 11   | DecayHold | Decay 31→0, hold at 31 (inverted) |
| 12   | Saw12 | Continuous sawtooth up: 0→31→0→31… |
| 13   | AttackHold | Attack 0→31, hold at 0 |
| 14   | Triangle14 | Triangle up-down: 0→31→0→31… |
| 15   | Hold31 | Same as types 4–7 |

All 8 shapes are verified against `pascal-baselines/envelope_shapes.json`
(64 amplitude steps per shape).

### Envelope clock

The envelope counter is 64-bit (split as two u32 words).  When the high word
overflows, `step_envelope()` is called and the amplitude advances by ±1 (or
resets for sawtooth continuations).

```
envelope_period = registers.envelope  (16-bit, up to 65535)
ticks per step  = envelope_period × ... (see AyConfig::delay_in_tiks)
```

The `first_period` flag distinguishes the initial decay/attack sweep from
subsequent holds or repeating sawtooth cycles.

## Mixer and amplitude

`synthesizer_mixer_q()` runs once per audio sample and accumulates stereo
output:

```
for each channel X in {A, B, C}:
    k = 1
    if tone enabled on X:   k &= ton_x        // gated by square wave
    if noise enabled on X:  k &= noise_val     // gated by noise bit
    if k == 0: contribute 0

    if envelope_en_x:
        amp_index = ampl          // 0–31 from envelope
    else:
        amp_index = amplitude_x * 2 + 1  // 0–15 → 1,3,5,…,31

    gain_l, gain_r = level_tables[amp_index]   // pre-calculated DAC values
    lev_l += gain_l * k
    lev_r += gain_r * k
```

The per-channel left/right balance is set by six panning indices in
`AyConfig` (`index_al`, `index_ar`, `index_bl`, …).

### Amplitude tables

Two sets of amplitude→DAC mapping tables are compiled at startup by
`calculate_level_tables()` (ported from `legacy/AY.pas`):

**AY table (16 entries, measured on real silicon):**
```
0, 836, 1212, 1773, 2619, 3875, 5397, 8823,
10392, 16706, 23339, 29292, 36969, 46421, 55195, 65535
```

**YM table (32 entries):** Finer-grained measurements by Hacker KAY; used
when the chip is configured as a YM2149F.

The tables are verified against `pascal-baselines/pt3_vol.json`.

## Synthesizer (`synth.rs`)

The `Synthesizer` struct wraps up to two `SoundChip` instances and exposes
the high-level API used by the playback engine and audio player.

### Rendering pipeline

```
Playback engine
    │  AyRegisters (tone, noise, mixer, amplitude, envelope) — once per frame
    ▼
Synthesizer::apply_registers(chip_idx, regs)
    │  Writes 14 register values into SoundChip
    ▼
Synthesizer::render_frame(n_samples)
    │  Loop n_samples times:
    │    for each chip:
    │      SoundChip::synthesizer_logic_q()  ← advance all counters 1 AY tick
    │      SoundChip::synthesizer_mixer_q()  ← accumulate stereo contribution
    │    FIR low-pass filter (optional)
    │    Clamp to i16 → StereoSample
    ▼
output_buf: Vec<StereoSample>   (pushed to audio ring buffer)
```

### FIR low-pass filter

A windowed-sinc filter with a Hanning window is applied to prevent aliasing.
Coefficients are pre-calculated by `calc_fir_coefficients()`:

```
cutoff_fraction = 0.45   (45 % of Nyquist)
for n in 0..filt_m:
    h[n] = sinc(2π × fc × (n - m/2)) × hanning(n, m)
```

Default filter order (`filt_m`) is 32 taps.  Each sample passes through a
circular delay-line convolution.

### Configuration (`AyConfig`)

| Parameter | Default | Description |
|-----------|---------|-------------|
| `sample_rate` | 48 000 Hz | Output sample rate |
| `ay_freq` | 1 773 400 Hz | AY chip clock |
| `interrupt_freq` | 50 Hz | Tracker frame rate |
| `is_filt` | `true` | Enable FIR filter |
| `filt_m` | 32 | Filter order |
| `global_volume` | 1.0 | Master gain (linear) |

Derived values:
- `ay_tiks_in_interrupt()` = `ay_freq / interrupt_freq / 8` ≈ 4 434 ticks/frame
- `sample_tiks_in_interrupt()` = `sample_rate / interrupt_freq` = 960 samples/frame
- `delay_in_tiks()` = fixed-point AY ticks per audio sample

## Audio output (`crates/vti-audio`)

```
Synthesizer::output_buf
    │  Vec<StereoSample> (i16 pairs)
    ▼
AudioPlayer::push_samples(&[StereoSample])
    │  Writes into RingBuf (circular queue, ~500 ms capacity)
    ▼
cpal data callback (audio device thread)
    │  Pops samples: i16 → f32 (-1.0..1.0)
    │  On underrun: output silence (0.0)
    ▼
OS audio driver → speakers / headphones
```

### Ring buffer

A fixed-capacity circular queue protects against timing jitter between the
render thread (writes) and the audio callback (reads).  Capacity is set to
`sample_rate / 2 + 1` (≈ 24 001 stereo samples at 48 kHz = 500 ms).

If the buffer is full when the render thread tries to push, the sample is
silently dropped (prevents render thread blocking).  If the buffer is empty
when the audio callback requests a sample, a silent frame (0, 0) is output
(prevents callback blocking / pops).

### Diagnostics

`AudioDiagnostics` counters available for monitoring:
- `pushed_samples` — total samples written by render thread
- `popped_samples` — total samples consumed by audio device
- `underrun_frames` — count of silent frames due to empty buffer
- `callback_count` — number of times cpal invoked the data callback

## Pascal baseline tests

The correctness of the chip emulator is verified by comparing Rust output
against JSON fixtures generated by running the original Pascal code through
FPC:

| Fixture | Coverage |
|---------|---------|
| `noise_lfsr.json` | 200-step LFSR sequence |
| `envelope_shapes.json` | All 8 shapes × 64 amplitude steps |
| `pt3_vol.json` | 16×16 PT3 volume table |
| `pattern_play_basic.json` | AY register values over 20 ticks |
| `pattern_play_envelope.json` | Same with envelope type 8 active |

Run them with:
```sh
cargo test -p vti-ay --test pascal_baseline_tests
```

To regenerate from Pascal source (requires FPC):
```sh
bash pascal-tests/run_harness.sh
```
